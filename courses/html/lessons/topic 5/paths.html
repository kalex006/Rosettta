5.2 Relative vs Absolute Pathing

Navigational Geometry: Mapping the relationship between resources and the document root.

The Pathing Paradox

In a "Zero-Backend" architecture like ROSETTA, understanding how files find each other is the most important technical skill. A path is not just a string of text; it is a directional instruction to the browser's fetch engine.

There are three primary methods for defining the location of an artifact: Absolute, Relative, and Root-Relative.

1. Absolute Paths (The Global Address)

An absolute path provides the full URL, including the protocol (http/https) and the domain name.

Usage: Use these only when linking to external websites or CDNs. Avoid using them for your own internal files, as it makes your project hard to test locally.

HTML absolute_example.html 
<!-- Points to a specific location on the internet -->
<a href="https://github.com/kalex/rosetta">View Source</a>

2. Relative Paths (The Local Relationship)

Relative paths describe the location of a file relative to the current document. This is the most portable way to build a website.

  • filename.html : Look in the same folder.
  • images/logo.png : Look in a subfolder named 'images'.
  • ../style.css : Go up one level, then look for the file.
  • ../../index.html : Go up two levels.
HTML relative_logic.html 
<!-- If we are in topic-05/5-2.html, we go up two levels to find style.css -->
<link rel="stylesheet" href="../../style.css">

<!-- To find 5-1.html in the same folder -->
<a href="5-1-anchors.html">Previous Lesson</a>

3. Root-Relative Paths (The Core Reference)

A path starting with a forward slash (/) tells the browser to start looking from the Domain Root.

Architect's Warning: On GitHub Pages, your project is often hosted in a subfolder (e.g., username.github.io/rosetta/). A root-relative path like /style.css will look for the file at the domain root, skipping your project folder entirely and resulting in a 404 error. Stick to Relative Paths for GitHub Pages compatibility.

The "Base" Element

If you find yourself writing ../../ too often, you can use the <base> element in the <head>. This sets a "starting point" for all relative URLs on the page.

HTML base_tag.html 
<head>
    <!-- All relative links now start from this directory -->
    <base href="https://kalex.github.io/rosetta/">
</head>

Best Practices for ROSETTA

To ensure ROSETTA stays "Zero-Backend" and portable:

  • Use Relative Paths for all internal documentation links.
  • Always use lowercase for file and folder names (some servers are case-sensitive, others are not).
  • Avoid spaces in filenames; use hyphens (-) instead.
